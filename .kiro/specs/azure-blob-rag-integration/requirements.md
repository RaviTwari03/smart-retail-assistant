# Requirements Document: Azure Blob Storage RAG Integration

## Introduction

The Smart Retail Assistant currently loads knowledge base documents from local files in the `./knowledge_base` directory. This feature integrates Azure Blob Storage as the centralized source of truth for the RAG (Retrieval-Augmented Generation) knowledge base, enabling scalable document management, multi-instance deployment support, and cloud-native architecture alignment. The system will maintain backward compatibility with the existing RAG pipeline while replacing local file operations with Azure Blob Storage operations.

## Glossary

- **Azure Blob Storage**: Microsoft Azure's cloud object storage service for unstructured data
- **BlobServiceClient**: Azure SDK client for managing blob storage accounts
- **ContainerClient**: Azure SDK client for managing containers within a blob storage account
- **Knowledge Base**: Collection of PDF and text documents used for RAG context retrieval
- **RAG Pipeline**: Retrieval-Augmented Generation pipeline that loads documents, generates embeddings, and performs similarity search
- **Vector Database**: ChromaDB instance storing document embeddings for semantic search
- **Document Chunk**: Segment of text created by splitting documents for embedding generation
- **Blob**: Individual file object stored in Azure Blob Storage
- **Connection String**: Azure credential string containing storage account name and access key
- **Container**: Logical grouping of blobs within Azure Blob Storage (equivalent to a bucket)
- **Temporary File**: Local file downloaded from blob storage for processing, deleted after use
- **Embedding**: Vector representation of text generated by sentence-transformers model

## Requirements

### Requirement 1: Azure Blob Storage Service Integration

**User Story:** As a system administrator, I want the application to connect to Azure Blob Storage using environment variables, so that credentials are securely managed and the system can access cloud-hosted documents.

#### Acceptance Criteria

1. WHEN the application starts, THE BlobServiceClient SHALL initialize using the AZURE_STORAGE_CONNECTION_STRING environment variable
2. WHEN the application starts, THE ContainerClient SHALL initialize using the AZURE_BLOB_CONTAINER environment variable (default: "knowledge-base")
3. IF the AZURE_STORAGE_CONNECTION_STRING environment variable is missing, THEN THE application SHALL raise a descriptive error indicating the missing configuration
4. IF the AZURE_BLOB_CONTAINER environment variable is missing, THEN THE application SHALL use the default container name "knowledge-base"
5. WHEN the BlobServiceClient is initialized, THE connection SHALL be validated by attempting to access the container
6. IF the container does not exist in Azure Blob Storage, THEN THE application SHALL raise a descriptive error indicating the missing container

### Requirement 2: List Blob Documents

**User Story:** As a developer, I want to retrieve a list of all documents stored in Azure Blob Storage, so that I can verify available knowledge base documents and manage document inventory.

#### Acceptance Criteria

1. WHEN the list_documents() function is called, THE function SHALL retrieve all blob names from the configured Azure Blob Storage container
2. WHEN blobs are retrieved, THE function SHALL return a list of blob names as strings
3. WHEN the container is empty, THE function SHALL return an empty list
4. IF an Azure SDK exception occurs during retrieval, THEN THE function SHALL log the error and raise a custom exception with a descriptive message
5. WHEN blobs are listed, THE function SHALL filter out any system files or metadata blobs (e.g., blobs starting with ".")

### Requirement 3: Upload Document to Azure Blob Storage

**User Story:** As a knowledge base manager, I want to upload documents to Azure Blob Storage through the API, so that I can add new knowledge base documents without manual blob storage access.

#### Acceptance Criteria

1. WHEN the upload_document(file_path, blob_name) function is called with a valid local file path, THE function SHALL upload the file to Azure Blob Storage with the specified blob name
2. WHEN a file is uploaded, THE function SHALL return the blob name and upload status
3. IF the local file does not exist, THEN THE function SHALL raise a FileNotFoundError with a descriptive message
4. IF the blob name already exists in the container, THEN THE function SHALL overwrite the existing blob
5. IF an Azure SDK exception occurs during upload, THEN THE function SHALL log the error and raise a custom exception with a descriptive message
6. WHEN a file is uploaded, THE function SHALL support PDF and text file formats

### Requirement 4: Download Document from Azure Blob Storage

**User Story:** As a RAG pipeline component, I want to download documents from Azure Blob Storage to temporary local storage, so that I can process documents for embedding generation.

#### Acceptance Criteria

1. WHEN the download_document(blob_name, local_path) function is called with a valid blob name, THE function SHALL download the blob from Azure Blob Storage to the specified local path
2. WHEN a blob is downloaded, THE function SHALL return the local file path
3. IF the blob does not exist in the container, THEN THE function SHALL raise a BlobNotFoundError with a descriptive message
4. IF an Azure SDK exception occurs during download, THEN THE function SHALL log the error and raise a custom exception with a descriptive message
5. WHEN a blob is downloaded, THE function SHALL create parent directories if they do not exist

### Requirement 5: Delete Document from Azure Blob Storage

**User Story:** As a knowledge base manager, I want to delete documents from Azure Blob Storage through the API, so that I can remove outdated or incorrect knowledge base documents.

#### Acceptance Criteria

1. WHEN the delete_document(blob_name) function is called with a valid blob name, THE function SHALL delete the blob from Azure Blob Storage
2. WHEN a blob is deleted, THE function SHALL return a deletion status confirmation
3. IF the blob does not exist in the container, THEN THE function SHALL raise a BlobNotFoundError with a descriptive message
4. IF an Azure SDK exception occurs during deletion, THEN THE function SHALL log the error and raise a custom exception with a descriptive message

### Requirement 6: Refactored Vector Database Creation from Azure Blobs

**User Story:** As a system administrator, I want the vector database creation process to load documents from Azure Blob Storage instead of local files, so that the knowledge base is centralized and scalable.

#### Acceptance Criteria

1. WHEN create_vector_db() is called, THE function SHALL retrieve all blob names from Azure Blob Storage
2. WHEN blobs are retrieved, THE function SHALL download each blob to a temporary directory
3. WHEN a blob is downloaded, THE function SHALL extract text from PDF files using PyPDFLoader
4. WHEN a blob is downloaded, THE function SHALL load text from TXT files using TextLoader
5. IF a blob cannot be processed, THEN THE function SHALL log the error and continue processing remaining blobs
6. WHEN documents are loaded, THE function SHALL split documents into chunks using RecursiveCharacterTextSplitter with chunk_size=300 and chunk_overlap=50
7. WHEN chunks are created, THE function SHALL generate embeddings using HuggingFaceEmbeddings with model "sentence-transformers/all-MiniLM-L6-v2"
8. WHEN embeddings are generated, THE function SHALL store vectors in ChromaDB at the configured persist_directory
9. WHEN vector database creation completes, THE function SHALL delete all temporary files and directories
10. WHEN vector database creation completes, THE function SHALL log the total number of documents loaded and chunks created

### Requirement 7: RAG Pipeline Integration with Azure Blobs

**User Story:** As a RAG component, I want the search_documents() function to use the vector database created from Azure Blob Storage documents, so that similarity search returns relevant knowledge base content.

#### Acceptance Criteria

1. WHEN search_documents(query) is called, THE function SHALL load the ChromaDB vector database from the configured persist_directory
2. WHEN a query is provided, THE function SHALL perform similarity search with k=3 results
3. WHEN search results are retrieved, THE function SHALL return the page_content of matching documents
4. IF the vector database does not exist, THEN THE function SHALL raise a descriptive error indicating the database needs to be created
5. IF an exception occurs during search, THEN THE function SHALL log the error and return an empty list

### Requirement 8: FastAPI Endpoint - List Blob Documents

**User Story:** As a frontend developer, I want an API endpoint to retrieve all available documents in Azure Blob Storage, so that I can display the knowledge base inventory in the UI.

#### Acceptance Criteria

1. WHEN a GET request is made to /blob-documents, THE endpoint SHALL return a list of all blob names in Azure Blob Storage
2. WHEN blobs are retrieved, THE response SHALL include a "status" field with value "success"
3. WHEN blobs are retrieved, THE response SHALL include a "documents" field containing the list of blob names
4. IF an error occurs during retrieval, THE response SHALL include a "status" field with value "error" and a "message" field with the error description
5. WHEN the endpoint is called, THE response HTTP status code SHALL be 200 for success and 500 for errors

### Requirement 9: FastAPI Endpoint - Upload Document

**User Story:** As a knowledge base manager, I want an API endpoint to upload documents to Azure Blob Storage, so that I can add new knowledge base documents through the web interface.

#### Acceptance Criteria

1. WHEN a POST request is made to /upload-document with a file, THE endpoint SHALL upload the file to Azure Blob Storage
2. WHEN a file is uploaded, THE response SHALL include a "status" field with value "success"
3. WHEN a file is uploaded, THE response SHALL include a "blob_name" field with the uploaded blob name
4. IF the file is not provided, THEN THE response SHALL include a "status" field with value "error" and a "message" field indicating the missing file
5. IF an error occurs during upload, THE response SHALL include a "status" field with value "error" and a "message" field with the error description
6. WHEN the endpoint is called, THE response HTTP status code SHALL be 200 for success and 400 for validation errors and 500 for server errors

### Requirement 10: FastAPI Endpoint - Delete Document

**User Story:** As a knowledge base manager, I want an API endpoint to delete documents from Azure Blob Storage, so that I can remove outdated knowledge base documents through the web interface.

#### Acceptance Criteria

1. WHEN a DELETE request is made to /delete-document/{blob_name}, THE endpoint SHALL delete the blob from Azure Blob Storage
2. WHEN a blob is deleted, THE response SHALL include a "status" field with value "success"
3. WHEN a blob is deleted, THE response SHALL include a "message" field confirming the deletion
4. IF the blob does not exist, THEN THE response SHALL include a "status" field with value "error" and a "message" field indicating the blob was not found
5. IF an error occurs during deletion, THE response SHALL include a "status" field with value "error" and a "message" field with the error description
6. WHEN the endpoint is called, THE response HTTP status code SHALL be 200 for success and 404 for not found and 500 for server errors

### Requirement 11: Error Handling and Validation

**User Story:** As a system operator, I want comprehensive error handling and validation throughout the Azure Blob Storage integration, so that issues are clearly reported and the system fails gracefully.

#### Acceptance Criteria

1. WHEN a connection string is missing, THE system SHALL raise an error with message "AZURE_STORAGE_CONNECTION_STRING environment variable is not set"
2. WHEN a container does not exist, THE system SHALL raise an error with message "Azure Blob Storage container '{container_name}' does not exist or is not accessible"
3. WHEN a blob is not found, THE system SHALL raise an error with message "Blob '{blob_name}' not found in container '{container_name}'"
4. WHEN an Azure SDK exception occurs, THE system SHALL log the full exception details including error code and message
5. WHEN a file operation fails, THE system SHALL log the operation type, file path, and error details
6. WHEN temporary files cannot be deleted, THE system SHALL log a warning but continue execution
7. WHEN an error occurs in the RAG pipeline, THE system SHALL log the error and return appropriate error responses to API clients

### Requirement 12: Comprehensive Logging

**User Story:** As a system administrator, I want detailed logging of all Azure Blob Storage operations, so that I can troubleshoot issues and monitor system behavior.

#### Acceptance Criteria

1. WHEN the BlobServiceClient is initialized, THE system SHALL log "Initializing Azure Blob Storage connection"
2. WHEN the ContainerClient is initialized, THE system SHALL log "Initializing Azure Blob Storage container: {container_name}"
3. WHEN blobs are listed, THE system SHALL log "Retrieved {count} blobs from container"
4. WHEN a blob is uploaded, THE system SHALL log "Uploading blob: {blob_name}"
5. WHEN a blob is downloaded, THE system SHALL log "Downloading blob: {blob_name} to {local_path}"
6. WHEN a blob is deleted, THE system SHALL log "Deleting blob: {blob_name}"
7. WHEN vector database creation starts, THE system SHALL log "Starting vector database creation from Azure Blobs"
8. WHEN documents are loaded, THE system SHALL log "Loaded {count} documents from {blob_count} blobs"
9. WHEN chunks are created, THE system SHALL log "Created {count} document chunks"
10. WHEN vector database creation completes, THE system SHALL log "Vector database created successfully at {persist_directory}"
11. WHEN temporary files are cleaned up, THE system SHALL log "Cleaned up temporary files from {temp_directory}"

### Requirement 13: Temporary File Management

**User Story:** As a system administrator, I want temporary files downloaded from Azure Blob Storage to be properly managed and cleaned up, so that disk space is not wasted and security is maintained.

#### Acceptance Criteria

1. WHEN blobs are downloaded for processing, THE system SHALL create a temporary directory with a unique name
2. WHEN blobs are downloaded, THE system SHALL store them in the temporary directory
3. WHEN vector database creation completes, THE system SHALL delete all files in the temporary directory
4. WHEN vector database creation fails, THE system SHALL still attempt to delete temporary files
5. IF temporary files cannot be deleted, THE system SHALL log a warning with the file path and error details
6. WHEN temporary directories are created, THE system SHALL use Python's tempfile module for secure creation

### Requirement 14: Azure Blob Storage Dependency

**User Story:** As a developer, I want the azure-storage-blob package to be included in the project dependencies, so that the Azure SDK is available for the integration.

#### Acceptance Criteria

1. WHEN requirements.txt is reviewed, THE azure-storage-blob package SHALL be listed with version 12.25.1 or later
2. WHEN the application starts, THE azure-storage-blob package SHALL be importable without errors
3. WHEN the application is deployed, THE azure-storage-blob package SHALL be installed in the deployment environment

### Requirement 15: Backward Compatibility

**User Story:** As a system operator, I want the existing RAG pipeline to continue functioning with Azure Blob Storage, so that the transition is seamless and no functionality is lost.

#### Acceptance Criteria

1. WHEN search_documents(query) is called, THE function SHALL return results in the same format as before (list of strings)
2. WHEN the /search-documents endpoint is called, THE response format SHALL remain unchanged
3. WHEN the vector database is created, THE ChromaDB storage format SHALL remain unchanged
4. WHEN the application starts, THE existing /search-documents endpoint SHALL continue to function without modification

## Acceptance Criteria Summary

This requirements document defines 15 comprehensive requirements covering:
- Azure Blob Storage service initialization and validation (Req 1)
- Document management operations: list, upload, download, delete (Req 2-5)
- Vector database creation from Azure Blobs (Req 6)
- RAG pipeline integration (Req 7)
- Three new FastAPI endpoints for document management (Req 8-10)
- Error handling and validation (Req 11)
- Comprehensive logging (Req 12)
- Temporary file management (Req 13)
- Dependency management (Req 14)
- Backward compatibility (Req 15)

All requirements follow EARS patterns and INCOSE quality rules, with explicit acceptance criteria that are testable and measurable.
